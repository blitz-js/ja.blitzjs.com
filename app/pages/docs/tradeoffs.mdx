何かを得ると、別の何かを失うと言われています。Blitz の場合、どうでしょうか
？

## フレームワークのロックイン {#framework-lock-in}

フレームワークと言えば、ある程度のロックインが必ずあります。React でアプリ
を作ったら、React にすごくロックインになります。

Next.js より Blitz のロックインは一つレベルの上です。Next.js のコードベー
スと比べて、Blitz のは*そんなに*違わないですが、ある違いでまだ簡単に
Next.js に戻れることができないです。Blitz で「eject」のような機能はないで
す。例えば、Blitz を使うとして、Blitz の認証を使うかもしれませんが、Blitz
の認証は Blitz に特有のものです。しかし、Next Auth に切り替えれば、Next.js
と同等の使用感が得られます。

Blitz の「zero-API data layer」も Blitz に特有のものです。したがって
、Next.js に切り替えれば、API 層を自分で管理するのが必要になります。（最初
から Next.js で開発したら、当然にこれはそうなので、特に失うことはないです
。）だが、バックエンドのクエリーロジックとミューテーションロジックが直接に
ポートできます。

## Next.js のアップグレード {#next-js-upgrades}

一番最新の Next.js リリースを使う希望があったら、Blitz が大抵ちょっとだけ
の古い版を使っていることを気になってください。Next.js が新しいバージョンを
出来ると、一般的にリリースの一周間内で Blitz をアップデートされます。

Yarn 使っている方、package.json である「resolutions」項目を使えば、先に
Next.js をアップグレードすることは可能です。

## マルチクライエント API {#multi-client-api}

「最終的に GraphQL にしなければいけないので、最初から GraphQL の方がいいで
しょう」と言う人がいます。ほとんどのアプリケーションは、この規模まで成長し
ないと考えていますが、マルチクライエントを使える API 対応することはまだ大
事な質問です。

多くのウェブアプリにとって、モバイルアプリが遅かれ早かれ必要になります
。Blitz はこう言うプロセスをできるだけ簡単にしたい希望があります。

そのため、GraphQL サーバーを Blitz に追加ことができます。いいチョイスなの
で、数人の利用者がこう言う風に Blitz を使っています。

## 高度なバックエンドアーキテクチャ {#advanced-backend-architecture}

現在、Nest.js や AdonisJs などと比べると、Blitz はバックエンドのアーキテク
チャが非常にシンプルです。その２つに 似ているパターンが Blitz で使えるので
すが、自分で実装するのは必要です。

事実、Blitz の中で必要なら、Nest.js を使えますが、デフォルトでもオプトイン
機能でも Blitz のアーキテクチャをレベルアップすることに興味があります。そ
の[話](https://github.com/blitz-js/blitz/discussions/1841)は今進んでいるの
で、ぜひ意見を出してください。

## サーバーレスじゃない場合、シングルスレッドで動き {#single-threaded-when-not-deployed-serverless}

この失うことは特定に Next.js の失うことです。Next.js はシングルスレッドで
できたアプリなので、ものすごく処理があれば、全てのリクエストがどんどん遅く
なるとかパフォマンス的にハズレになります。

これを解決するには、バックグラウンド処理を他のプロセスに分離することです。
複数のバックグラウンドプロセスを実行することはそれほど難しいことではないで
すが、できるだけ簡単になるためにドキュメントや API を追加したいと考えてい
ます。
